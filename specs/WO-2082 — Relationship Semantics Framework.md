## SOURCE GROUNDING (read this before the spec — it corrects an earlier oversold pitch)

Earlier in this session, a wall of externally-generated analysis proposed a large new subsystem
("Unified Causal Projection Kernel" / "Projection Constraint System" / WO-2094) based on a paper
at a `tandfonline.com` URL, scored "9.1/10," citing invented KRYLO subsystems ("Structural
Coherence Engine," "Convergence Engine," "Drift Monitor") that do not exist anywhere in this
codebase. That proposal was not built or filed — see [[feedback_verify_patches_against_real_files]].

The Founder then supplied the actual paper as a local PDF: `docs/jds1211.pdf` —
**Wang, Richardson & Robins, "Causal Inference: A Tale of Three Frameworks," Journal of Data
Science 24(1), 53–85, January 2026, DOI 10.6339/25-JDS1211.** Confirmed by reading pages 1–5
(title, authors, abstract, intro) directly. This is a **comparative review/tutorial paper**
("aimed at researchers and graduate students... seeking a conceptual foundation"), written by
three of the actual founders of these frameworks (Robins: potential outcomes / g-methods;
Richardson: SWIGs). It is NOT a paper introducing new completeness proofs or a "meta-constraint
on representation" as the earlier pitch claimed — it is a well-grounded synthesis of
**already-established** connections between three ways of formalizing a causal claim:

1. **Potential Outcomes** — a relationship expressed as an intervention/counterfactual effect
   (what would happen to Y if A were set to a different value)
2. **Structural Equation Models (NPSEMs)** — a relationship expressed as a mechanism/function
   (Y is generated by a specific functional dependency on its causes)
3. **Causal DAGs / SWIGs** — a relationship expressed as a structural/independence constraint
   (an edge encodes "not independent of," not a mechanism or an effect size)

The paper's real, load-bearing point for KRYLO: **these three are different claims about what an
edge means, not interchangeable labels for "A causes B."** Treating them as equivalent is a
documented, well-known failure mode in applied causal reasoning. That is directly relevant to
this WO's actual job, scoped narrowly below — it is NOT grounds for a new subsystem, ledger, or
kernel. No new module is being introduced by this WO.

## 1. SINGLE RESPONSIBILITY CHECK

**Job:** Give every edge in a Domain Package's `causalMap` (the fourth required output field,
locked in `domainpackage.js:17`, WO-2064) an explicit, typed meaning — answering "what kind of
relationship is this?" — instead of leaving `causalMap`'s internal shape completely
unconstrained, which is its current state (confirmed: `causalMap` is required to exist as a
top-level key, `domainpackage.js:17,51`, but nothing validates or types what's inside it).

**Output:** A typed edge schema + a validator that domain implementations must satisfy when they
populate `causalMap` before sealing (`emitDomainOutput`, `domainpackage.js:78`).

---

## 2. BOUNDARY DECLARATION

**Input contract:** An unsealed Domain Package's proposed `output.causalMap` — an array of edge
objects, each connecting two identifiers within that domain's own `componentGraph` or
`domainState`.

**Output contract:** Same array, validated against the typed edge schema below. Invalid edges
(missing/unknown `relationType`, or a `MECHANISTIC` edge missing its function description, etc.)
cause `validateDomainOutput` to reject the whole output, exactly as it already does for missing
required keys and invariant leakage (`domainpackage.js:36-72`).

**Explicit exclusions:**
- Does NOT compute, infer, or discover relationships. Domain implementations declare their own
  edges; this WO only types and validates the declaration.
- Does NOT touch Decision Invariants (WO-2063), Metric Adapter (WO-2066), Availability Filter
  (WO-2068), or Cone Surface (WO-2067). Those phases run after sealing and are unaffected.
- Does NOT introduce cross-domain reconciliation, divergence tracking, arbitration, or any new
  engine/subsystem. One domain's `causalMap` is validated in isolation, same as today.
- Does NOT require every domain to populate every relation type. An empty `causalMap` remains
  valid (as it is today) — this WO only constrains the *shape* of edges when they exist.

## 4. STRATEGIC LEVERAGE STATEMENT

**Statement:** Prevents KRYLO's causal layer from silently conflating "these two things are
correlated," "this mechanism generates that value," and "this intervention measurably changes
that outcome" into one undifferentiated "edge" — which would corrupt downstream trust in
`causalMap` before it has any real consumer, rather than after (the cheapest possible point to
fix this).

---

## 5. OUTPUT GRAVITY

**"The single thing this WO produces that matters most is"** a required `relationType` field on
every `causalMap` edge, drawn from a closed, three-value vocabulary, so no edge can claim an
unstated kind of causal relationship.

---

## 6. FORMULA / CONTRACT

**Edge schema (`causalMap[i]`):**

```
{
  from:         string,          // identifier within this domain's componentGraph/domainState
  to:           string,          // identifier within this domain's componentGraph/domainState
  relationType: 'MECHANISTIC' | 'STRUCTURAL' | 'INTERVENTIONAL',
  description:  string,          // human-readable, domain-native — required for all types
  // type-specific required fields:
  mechanism:    string | null,   // required if MECHANISTIC — the functional dependency claimed
  effect:       { magnitude: number|null, direction: 'UP'|'DOWN'|'NEUTRAL'|null } | null,
                                  // required if INTERVENTIONAL
}
```

- `MECHANISTIC` — claims `to` is generated by a function of `from` (SCM-style). Requires
  `mechanism` to be non-empty.
- `STRUCTURAL` — claims `from` and `to` are not independent, with no mechanism or effect size
  claimed (DAG-style). No additional required fields beyond `description`.
- `INTERVENTIONAL` — claims a measured or estimated effect of changing `from` on `to` (PO-style).
  Requires `effect.direction` to be set.

**Validation rule (added to `validateDomainOutput`, `domainpackage.js`):** for each entry in
`output.causalMap`, reject if `relationType` is missing or not one of the three values, or if
the type-specific required field for that `relationType` is missing.

**Normalization:** not applicable — this is a semantic/structural contract, not a numeric signal;
§16's 0–100 scale does not apply to edge typing.
